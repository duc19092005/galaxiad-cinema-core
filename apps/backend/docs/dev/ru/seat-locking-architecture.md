# Seat Locking Architecture — Запись Решения по Проектированию

> **Статус документа:** Завершен (V1)
> **Последнее обновление:** 2026-06-30

---

## 1. Постановка задачи

Нам нужен **механизм блокировки мест в реальном времени** в процессе бронирования билетов. Когда посетитель выбирает места и начинает оформление заказа, эти места должны быть временно зарезервированы, чтобы другие посетители не могли их выбрать. Механизм должен:

1. **Блокировать мгновенно** — другие пользователи видят место занятым сразу
2. **Авто-освобождать** — места разблокируются, если оплата не завершена в течение таймаута
3. **Обрабатывать отключения** — закрытие вкладки браузера → места освобождаются
4. **Быть простым в обслуживании** — избегать избыточного проектирования для временной блокировки

---

## 2. Рассмотренные варианты

### Вариант A: SSE + HTTP POST ✅ (Выбран)

- **Сервер → Клиент:** SSE (Server-Sent Events) — нативный API браузера, однонаправленная передача
- **Клиент → Сервер:** HTTP POST — стандартный REST API
- **Хранение:** In-memory (ConcurrentDictionary в Singleton)

### Вариант B: SignalR (WebSocket)

- Двунаправленный канал связи
- Требует согласования WebSocket с резервными транспортами
- Нужен Redis backplane для масштабирования на несколько инстансов

### Вариант C: Polling (setInterval)

- Фронтенд опрашивает сервер каждые X секунд о состоянии блокировки
- Просто реализовать, но расточительно (избыточные сетевые запросы)
- Задержка обновлений (не в реальном времени)

### Вариант D: Блокировка в Базе Данных

- Состояние блокировки хранится в SQL Server
- Самый надежный, но самый медленный (круговой запрос к БД)
- Риск истощения пула соединений

---

## 3. Решение: SSE + HTTP POST

### Почему мы выбрали это

| Критерий | SSE + POST | SignalR | Polling | DB Lock |
|---------|-----------|---------|---------|---------|
| В реальном времени | ✅ Мгновенно | ✅ Мгновенно | ❌ С задержкой | ❌ Медленно |
| Сложность | ✅ Низкая | ❌ Высокая | ✅ Низкая | ❌ Средняя |
| Авто-переподключение | ✅ Встроено | ⚠️ Вручную | ✅ Естественно | N/A |
| CDN совместимость | ✅ Да | ⚠️ Частично | ✅ Да | ✅ Да |
| Сохранение состояния | ⚠️ RAM | ⚠️ RAM | ⚠️ DB/Redis | ✅ Надежно |
| Стоимость обслуживания | ✅ Очень низкая | ❌ Высокая | ✅ Низкая | ❌ Средняя |
| **Итого** | **✅ Лучший** | ❌ | ⚠️ | ❌ |

### Принятые компромиссы

| Компромисс | Почему приемлемо |
|-----------|-----------------|
| **Хранение в RAM** (теряется при перезагрузке) | Блокировки временные (макс 10 мин). Клиенты автоматически переподключаются и получают свежее состояние. |
| **SSE однонаправленный** | Преднамеренно — блокировка/разблокировка это отдельные REST-действия с немедленным HTTP-ответом (200/409). |
| **Нет Redis** | V1 работает на одном сервере. При масштабировании добавим Redis pub/sub. |

## 4. Обзор архитектуры

### Основные компоненты

```
┌──────────────────────────────────────────────────────┐
│                    SeatSseManager                      │
│                  (Singleton Service)                    │
│                                                        │
│  ┌─────────────────────────────────────────────────┐  │
│  │  _scheduleSeatLocks                              │  │
│  │  ConcurrentDictionary<scheduleId,                │  │
│  │    ConcurrentDictionary<seatId, (userName, clientId)>> │
│  └─────────────────────────────────────────────────┘  │
│                                                        │
│  ┌─────────────────────────────────────────────────┐  │
│  │  _scheduleSubscribers                           │  │
│  │  ConcurrentDictionary<scheduleId,                │  │
│  │    ConcurrentDictionary<subscriberId, callback>> │  │
│  └─────────────────────────────────────────────────┘  │
│                                                        │
│  ┌─────────────────────────────────────────────────┐  │
│  │  _scheduleEventCounters                         │  │
│  │  ConcurrentDictionary<scheduleId, int>           │  │
│  └─────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

### Потокобезопасность

Все структуры данных используют `ConcurrentDictionary` — потокобезопасны по умолчанию. Метод `LockSeat()` использует атомарный `TryAdd` для предотвращения состояний гонки при одновременной блокировке одного и того же места двумя пользователями.

### Поток данных

```
POST /seats/lock  ──►  LockSeat() ──►  TryAdd в словарь
                        │
                        ├─► Успешно? ──► BroadcastEvent("seat-locked")
                        │                     │
                        │              SSE подписчики получают обновление
                        │
                        └─► Ошибка? ──► Возврат 409 Conflict

SSE отключение ──►  ReleaseSeatsByClient()
                        │
                        └─► Для каждого места клиента:
                              TryRemove → BroadcastEvent("seat-unlocked")

Hangfire job ────►  ReleaseSeatsForSchedule()
                        │
                        └─► Для каждого seatId:
                              TryRemove → BroadcastEvent("seat-unlocked")
```

## 5. Детали протокола SSE

### Формат события

```
id: {eventId}
event: {eventType}
data: {jsonPayload}

```

### Типы событий

| Событие | Когда отправляется | Данные |
|--------|-------------------|--------|
| `initial-state` | Новый подписчик | `{ "lockedSeats": { "A1": "UserA" } }` |
| `seat-locked` | Кто-то заблокировал место | `{ "seatId": "A1", "userName": "UserA", "lockedSeats": {...} }` |
| `seat-unlocked` | Кто-то освободил место | `{ "seatId": "A1", "lockedSeats": {...} }` |

### Heartbeat

Строка комментария (`: heartbeat\n\n`) отправляется каждые 15 секунд для предотвращения закрытия соединения прокси/балансировщиком.

### Переподключение

`EventSource` (встроенный в браузер) автоматически обрабатывает переподключение. В V1 при каждом переподключении клиент получает полный снимок `lockedSeats` через `initial-state`.

## 6. Ограничения V1

| Ограничение | Влияние | Улучшение в будущем |
|------------|---------|-------------------|
| **Нет Redis backplane** | Разделение состояния при нескольких инстансах | Добавить Redis pub/sub |
| **Нет срока действия отдельных блокировок** | Блокировка остается в RAM при внезапном отключении клиента | Добавить фоновую задачу очистки |
| **Нет постоянного хранения** | Потеря блокировок при перезагрузке | Приемлемо (макс 10 мин) |
| **Память одного сервера** | RAM растет с количеством активных сеансов | Мониторинг и установка лимитов |

## 7. Сравнение SSE vs SignalR

| Аспект | SSE | SignalR |
|-------|-----|---------|
| Транспорт | Нативный HTTP (long-poll или stream) | WebSocket + fallbacks (SSE, long-polling) |
| Поддержка браузеров | Все современные браузеры | Требуется библиотека `@microsoft/signalr` |
| Авто-переподключение | Встроено (`EventSource`) | Настраиваемое |
| Макс. соединений (браузер) | 6 на домен (HTTP/1.1) | Безлимитно (WebSocket) |
| Бинарные данные | Нет (только текст) | Да |
| Пользовательские заголовки | Нет (через URL params) | Да (через negotiate) |
| Через прокси | Работает через HTTP прокси | Некоторые прокси блокируют WebSocket |
| CDN кэширование | Работает | WebSocket upgrade не удается |
| Сложность протокола | Простой (text/event-stream) | Сложный (negotiation, hub protocol) |
| Клиентская библиотека | Не нужна (встроена в браузер) | npm пакет `@microsoft/signalr` |
